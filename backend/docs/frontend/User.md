# Фронт: User/Auth API

Актуальный контракт для фронта по `v1` API пользователей, авторизации, подтверждению email и восстановлению пароля.

---

## Базовые вещи

- Base URL API: `https://api.vennro.ru/v1`
- Для всех запросов с телом использовать `Content-Type: application/json`
- Успешные ответы с данными идут в формате `{ "data": { ... } }`
- Успешные ответы без данных:
  - `204 No Content` для logout
  - `{ "data": { "success": 1 } }` для create/update/delete/confirm/reset flows
- Ошибки домена обычно приходят как `{ "error": { "code": number, "message": string } }`
- Ошибки валидации: `{ "validations": [{ "field": string, "message": string }] }`
- Для защищённых endpoint нужен заголовок `Authorization: Bearer <access_token>`

### Важное дополнение: Cookies и SSO
Для работы авторизации между поддоменами (например, `auth.vennro.ru` и `vennro.ru`) бэкенд использует **HttpOnly Cookies** для хранения `refresh_token`.
- Браузер сам сохраняет и отправляет `refresh_token`.
- JS-код не имеет доступа к этому токену (защита от XSS).
- Для всех запросов к API фронтенд должен использовать `{ credentials: 'include' }` в fetch или `withCredentials: true` в axios.

## 1. Регистрация

Создаёт пользователя и отправляет письмо с подтверждением email. После регистрации токены не выдаются.

- Метод: `POST`
- URL: `/v1/users/create`

Тело:

```json
{
  "lastName": "Иванов",
  "firstName": "Иван",
  "email": "user@example.com",
  "password": "Secret123!"
}
```

Успех:
- `201`

```json
{
  "data": {
    "success": 1
  }
}
```

Ошибки:
- `409` если email уже зарегистрирован
- `422` если не прошла валидация

Что делать на фронте:
- показать сообщение, что нужно подтвердить email
- не считать пользователя залогиненным

---

## 2. Подтверждение email

Письмо ведёт на фронтенд авторизации, например:

`{FRONTEND_AUTH_URL}/email-verification?token=...`

Фронт берёт `token` из query и отправляет его на бэкенд.

- Метод: `POST`
- URL: `/v1/auth/confirm-email`

Тело:

```json
{
  "token": "token-from-query"
}
```

Успех:
- `200`

```json
{
  "data": {
    "success": 1
  }
}
```

Ошибки:
- `409` если ссылка недействительна или истекла
- `422` если `token` не передан или пустой

---

## 3. Логин

Вход по email и паролю. Работает только для активного и не удалённого пользователя.

- Метод: `POST`
- URL: `/v1/auth/login`

Тело:

```json
{
  "email": "user@example.com",
  "password": "Secret123!"
}
```

Успех:
- `200`
- Устанавливается кука `refresh_token` (Domain: .vennro.ru, HttpOnly).

```json
{
  "data": {
    "access_token": "eyJ...",
    "expires_in": 900
  }
}
```

Значение полей:
- `access_token` использовать в `Authorization: Bearer ...`
- `expires_in` это TTL access token в секундах

Ошибки:
- `409` неверные учётные данные
- `409` email не подтверждён
- `409` аккаунт забанен
- `422` ошибки валидации

---

## 4. Обновление токенов

Получение новой пары токенов. `refresh_token` автоматически берется из куки.

- Метод: `POST`
- URL: `/v1/auth/refresh`
- Тело: *пустое* (обязателен заголовок `credentials: 'include'`)

Успех:
- `200`
- Обновляется кука `refresh_token`.

```json
{
  "data": {
    "access_token": "eyJ...",
    "expires_in": 900
  }
}
```

Важно:
- refresh token ротируется автоматически в куках.
- если куки нет или она невалидна — придет `401`.

Ошибки:
- `401` refresh token невалиден, истёк, отозван или пользователь больше не активен

Рекомендация для фронта:
- при `401` на обычном API-запросе можно один раз попробовать `/auth/refresh`
- если refresh тоже вернул `401`, удалить токены и отправить пользователя на логин

---

## 5. Logout

Logout отзывает текущий refresh token и удаляет куку.

- Метод: `POST`
- URL: `/v1/auth/logout`
- Тело: *пустое*

Успех:
- `204`
- тело пустое

Ошибки:
- `401` если вы уже не авторизованы (кука протухла или отсутствует)

Что делать на фронте:
- удалить локально `access_token`.
- сделать redirect на логин.

---

## 6. Восстановление пароля

Состоит из двух шагов.

### 6.1. Запрос письма на сброс

- Метод: `POST`
- URL: `/v1/auth/password-reset`

Тело:

```json
{
  "email": "user@example.com"
}
```

Успех:
- `201`

```json
{
  "data": {
    "success": 1
  }
}
```

Ошибки:
- `409` пользователь не найден или не активен
- `422` ошибка валидации

### 6.2. Подтверждение нового пароля

Фронтовая ссылка из письма ведёт на поддомен авторизации:

`{FRONTEND_AUTH_URL}/password-reset-confirm?token=...`

Тело запроса на сохранение нового пароля:

- Метод: `POST`
- URL: `/v1/auth/password-reset/confirm`

```json
{
  "token": "token-from-query",
  "password": "NewSecret123!"
}
```

Успех:
- `201`

```json
{
  "data": {
    "success": 1
  }
}
```

Ошибки:
- `409` токен недействителен или истёк
- `422` ошибка валидации пароля

Важно:
- после успешного `password reset confirm` бэкенд отзывает все refresh token пользователя
- значит после смены пароля старые login-сессии через refresh продолжать работу не смогут

## 7. Список пользователей

Защищённый endpoint для списка пользователей.

- Метод: `GET`
- URL: `/v1/users`
- Заголовок: `Authorization: Bearer <access_token>`

Query-параметры:
- `page`
- `perPage`
- `search`
- `dateFrom` в формате `Y-m-d`
- `dateTo` в формате `Y-m-d`

Успех:
- `200`

```json
{
  "data": {
    "count": 145,
    "items": [
      {
        "id": 1,
        "role": {
          "id": 1,
          "label": "Администратор"
        },
        "first_name": "Иван",
        "last_name": "Иванов",
        "email": "user@example.com",
        "avatar": null,
        "created_at": "2024-01-01 12:00:00",
        "is_active": true
      }
    ]
  }
}
```

Ошибки:
- `401` нет access token или он невалиден
- `422` ошибки в query-параметрах

## 8. Просмотр пользователя

Публичный endpoint, токен не нужен.

- Метод: `GET`
- URL: `/v1/users/{id}`

Успех:
- `200`

```json
{
  "data": {
    "id": 1,
    "role": {
      "id": 1,
      "label": "Администратор"
    },
    "first_name": "Иван",
    "last_name": "Иванов",
    "email": "user@example.com",
    "avatar": "https://storage.vennro.ru/vs-project/avatars/1/abc123.jpg",
    "created_at": "2024-01-01 12:00:00",
    "is_active": true
  }
}
```

Ошибки:
- `404` пользователь не найден или удалён

## 9. Обновление пользователя

Можно обновлять свой профиль, а администратор может обновлять и других пользователей.

- Метод: `PATCH`
- URL: `/v1/users/update/{id}`
- Заголовок: `Authorization: Bearer <access_token>`

Тело:

```json
{
  "lastName": "Петросян",
  "firstName": "Артем",
  "email": "vennro@gmail.com"
}
```

Успех:
- `200`

```json
{
  "data": {
    "success": 1
  }
}
```

Ошибки:
- `401` пользователь не авторизован
- `409` пользователь не найден
- `409` email уже занят
- `403`/`409` по доменной ошибке доступа, если нет прав
- `422` ошибка валидации

---

## 10. Удаление пользователя

Мягкое удаление пользователя.

- Метод: `DELETE`
- URL: `/v1/users/delete/{id}`
- Заголовок: `Authorization: Bearer <access_token>`

Успех:
- `200`

```json
{
  "data": {
    "success": 1
  }
}
```

Важно:
- при удалении пользователя его refresh token'ы отзываются
- уже выданный access token может дожить до конца своего TTL, но новые access token через refresh получить уже нельзя

Ошибки:
- `401` пользователь не авторизован
- `404` пользователь не найден
- `409` пользователь уже удалён
- `403`/`409` если нет прав на удаление

## 11. Загрузка аватара

Загрузка аватара пользователя. Можно загружать свой аватар, админ может менять аватар другого пользователя. Файл загружается потоково, сервер не хранит его в оперативной памяти.

- Метод: `POST`
- URL: `/v1/users/{id}/avatar`
- Заголовок: `Authorization: Bearer <access_token>`
- Content-Type: `multipart/form-data`
- Поле формы: `avatar` (файл)

Допустимые форматы: JPG, PNG, GIF, WEBP, HEIC/HEIF. Максимальный размер: 30 МБ.

Успех:
- `200`

```json
{
  "data": {
    "avatar": "https://storage.vennro.ru/vs-project/avatars/1/abc123def456.jpg"
  }
}
```

Ошибки:
- `401` не авторизован
- `403` нет прав
- `404` пользователь не найден
- `409` код 14 — недопустимый формат файла
- `409` код 15 — файл слишком большой
- `409` код 13 — файл не передан

Что делать на фронте:
- отобразить новый аватар по URL из ответа, так как ответ приходит в виде полного URL.
- бэкенд отдает полную ссылку, ничего дополнительно склеивать не нужно.
- при ошибке 409 показать соответствующее сообщение.

Удаление аватара пользователя.

## Краткий flow для фронта

1. Регистрация: `POST /v1/users/create`
2. Подтверждение email: `POST /v1/auth/confirm-email`
3. Логин: `POST /v1/auth/login`
4. Запросы к защищённым endpoint'ам с `Authorization: Bearer <access_token>` и обязательным `{ credentials: 'include' }`.
5. При `401` на защищённом API:
   - один раз попробовать `POST /v1/auth/refresh` (тело пустое, кука улетит сама)
   - при успехе сохранить новый `access_token` и повторить исходный запрос
   - при `401` на refresh сделать logout на фронте и отправить пользователя на страницу логина
6. Logout: `POST /v1/auth/logout` (тело пустое, кука удалится бэкендом)
7. Password reset:
   - `POST /v1/auth/password-reset`
   - `POST /v1/auth/password-reset/confirm`

## Переменные окружения бэкенда, которые важны фронту

- `FRONTEND_URL` — базовый URL главного сайта
- `FRONTEND_AUTH_URL` — базовый URL сервиса авторизации (для ссылок в письмах)
- `JWT_SECRET` — только для бэкенда, фронту не нужен, но auth без него не работает

Фронт должен иметь страницы:
- для подтверждения email
- для подтверждения сброса пароля

Обе страницы должны уметь взять `token` из query string и отправить его в соответствующий endpoint.
