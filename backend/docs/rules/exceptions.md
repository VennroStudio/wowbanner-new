# Исключения и HTTP-коды

Обработка исключений — в middleware (`src/Components/Http`).

---

## Таблица

| Ситуация | Исключение | HTTP |
|----------|------------|------|
| Токен невалиден / отозван / истёк | `AuthenticationException` | 401 |
| Нет прав на действие | `AccessDeniedException` | 403 |
| Ресурс не найден по ID | `NotFoundException` | 404 |
| Маршрут не найден | Slim сам | 404 |
| Метод не разрешён | Slim сам | 405 |
| Нарушено бизнес-правило | `DomainExceptionModule` | 409 |
| Данные не прошли валидацию | `ValidationException` (авто) | 422 |
| Неверный тип/формат аргумента | `InvalidArgumentException` | 400 |

---

## Когда какое исключение

- **`DomainExceptionModule`** — основное для бизнес-ошибок в Handler'ах. Принимает `module`, `message` (ключ перевода), `code` (числовой код). Возвращает 409 с переведённым сообщением.
- **`AuthenticationException`** — токен невалиден, отсутствует или истёк. Возвращает 401.
- **`AccessDeniedException`** — пользователь известен, но нет прав. Возвращает 403.
- **`NotFoundException`** — ресурс не найден по ID. Возвращает 404.
- **`InvalidArgumentException`** — технически неверный аргумент до входа в бизнес-логику. Возвращает 400.
- **`ValidationException`** — бросается автоматически из `Validator`, вручную не бросать. Возвращает 422.

---

## Denormalizer

- `denormalize()` — для HTTP-запросов (мягкие типы, лишние поля игнорируются)
- `denormalizeStrict()` — для внутренних вызовов (строгие типы)
- Автоматического преобразования `snake_case` → `camelCase` нет — имена полей JSON должны совпадать со свойствами DTO
- Ошибки денормализации → 422
